## Introduction

지난 9월 업스테이지에서 주최한 [Global AI week AI hackathon](/blog/Projects/Hackathon/2024-10-01-ai-hackathon-2024-upstage-global-ai-week-ai-hackathon/)에서 사용한 기술들을 다시금 정리하는 것이 좋을 것 같아 이와 같이 해당 기술들에 대한 포스팅을 가져왔습니다.

최근 LLM(대형 언어 모델) 기반 서비스를 개발할 때 가장 핵심이 되는 패턴은 단연 **RAG(Retrieval-Augmented Generation, 검색 증강 생성)입니다.** 모델 내부 지식에만 의존하는 '폐쇄형 시험(Closed-book Exam)' 방식에서 탈피하여, 외부 지식 베이스에서 필요한 정보를 찾아와 답변을 생성하는 '오픈북 시험(Open-book Exam)'으로의 전환은 환각 현상(Hallucination) 제어와 최신성 확보를 위한 필수 전략입니다.

그러나 실제 프로덕션 수준의 RAG 서비스를 구축하려다 보면 다음과 같은 현실적인 장벽에 부딪히게 됩니다.
1. 문서를 쪼개고, 임베딩을 생성하고, 벡터 데이터베이스에 저장한 뒤, 유저의 질문에 답변하기까지 너무 많은 컴포넌트가 얽혀 있어 순수 API로만 구현하면 스파게티 코드가 되기 쉽습니다.
2. 사용자의 질문이 모호하여 관련 문서 검색에 실패하거나 검색 품질이 좋지 않을 때, 기존의 단순 일방향(Linear) RAG 구조는 무조건 오답을 내거나 잘못된 정보를 참조합니다.

이 장벽들을 허물고 컴포넌트들을 규격화하여 조율(Orchestration)하기 위해 탄생한 프레임워크가 바로 **LangChain(랭체인)이며**, 더 나아가 검색 품질을 평가하여 피드백 루프를 돌리고 최선의 결과를 도출하기 위해 순환형 워크플로우를 정의하게 해주는 도구가 **LangGraph(랭그래프)입니다.** 

이번 글에서는 RAG 패러다임 위에 왜 LangChain과 LangGraph가 더해져야만 하는지, 그리고 Reranker를 도입하여 어떻게 RAG의 치명적인 한계를 돌파할 수 있는지 분석해 보겠습니다.

## 1. LangChain의 등장 배경과 필요성

LLM을 비즈니스 로직에 결합하기 위해 기초적인 API 호출 코드를 작성해 본 개발자라면 누구나 한 번쯤 "모델을 OpenAI에서 Anthropic의 Claude로 바꾸거나, 오픈소스 모델로 변경할 때 코드 규격을 매번 뜯어고쳐야 하는가?"라는 문제에 직면하게 됩니다.

LangChain은 이러한 파편화를 해결하기 위한 **'추상화(Abstraction) 레이어'를** 제공합니다.

### 1) ODBC / JDBC에 비유하는 추상화 레이어
과거 다양한 데이터베이스(Oracle, MySQL, MS SQL 등)가 등장했을 때, 각각의 커넥터를 하드코딩하던 문제를 해결하기 위해 표준 규격인 **ODBC(Open Database Connectivity)와** **JDBC**가 제안되었습니다. 덕분에 개발자는 데이터베이스의 내부 구현과 무관하게 공통 인터페이스로 쿼리를 보낼 수 있게 되었고, 비즈니스 코드를 거의 수정하지 않고도 DB를 교체할 수 있었습니다.

LangChain은 LLM 생태계의 ODBC/JDBC 역할을 자처합니다. 거의 모든 LLM API와 데이터 커넥터, 프롬프트 가공 방식 등을 표준 규격으로 감싸주기 때문에, API 키와 모델 선언부만 수정하면 하위 애플리케이션 코드를 유지한 채 모델이나 벡터 DB를 교체할 수 있는 유연성을 확보하게 됩니다.

### 2) 불필요한 세부 사항의 은폐
자동차 시동을 걸 때 엔진의 크랭크축이 돌아가고 연료가 분사되는 복잡한 내부 로직을 운전자가 알 필요 없이 단순 시동 버튼만 누르면 되듯이, LangChain은 텍스트 분할, 임베딩 차원 일치 계산, 컨텍스트 주입 템플릿 처리 등의 까다로운 가공 절차를 클래스 내부로 숨겨 줍니다. 이를 통해 개발자는 LLM을 조율하는 상위 논리(Workflow) 설계에만 전념할 수 있습니다.

### 💡 무상태(Stateless)의 극복: LLM이 대화의 흐름을 기억하는 원리
많은 입문자들이 챗봇을 구현할 때 흔히 하는 착각이 "LLM API가 이전의 질문과 답변을 기억하고 있을 것"이라는 점입니다. 하지만 모든 LLM API는 철저히 **무상태(Stateless)로** 동작합니다. 마치 사칙연산 계산기처럼, 새로운 입력이 오면 과거의 맥락은 완전히 지워집니다. 
따라서 대화형 서비스를 만들려면 개발자가 매번 **[이전 대화 내역 전체 + 새로운 질문]을** 수동으로 결합하여 LLM에 전달해야 합니다. LangChain의 `Memory` 모듈은 이 귀찮고 번잡한 맥락 병합 연산을 백그라운드에서 표준화된 포맷으로 가공해 주는 역할을 담당합니다.

### 🛠️ 구조적 깊이: LCEL 파이프 연산자(|)의 내부 동작 메커니즘
많은 개발자들이 LCEL(LangChain Expression Language)의 파이프 연산자(`|`)를 보고 단순한 문법적 설탕(Syntactic Sugar)이라고 생각합니다. 하지만 LCEL의 내부 동작은 전혀 다릅니다.
LCEL은 입력된 연결 고리를 바탕으로 내부적으로 **추상 구문 트리(AST, Abstract Syntax Tree)와** 유사한 형태의 데이터 흐름 그래프(Runnable Sequence)를 컴파일해 냅니다. 이 컴파일 과정 덕분에:
- 독립적인 실행 경로를 가진 서브 체인들이 존재할 경우, 엔진이 이를 감지하여 **자동으로 병렬 처리**합니다 (예: 여러 벡터 스토어에서 동시에 정보를 조회할 때 멀티스레딩이 기본 작동).
- 중간 노드들의 출력이 완성되는 즉시 다음 노드로 토큰을 흘려보내는 **스트리밍 프로토콜**을 기본적으로 보장합니다.
- 특정 모델 호출이 실패했을 때의 **자동 대체(Fallback) 메커니즘**을 선언적으로 결합할 수 있습니다.

## 2. LangChain의 핵심 모듈과 기술적 구성 요소

LangChain은 LLM 서비스를 구축하는 과정을 다음과 같이 추상화된 핵심 모듈로 세분화하여 관리합니다.

*   **Model I/O**:
    *   **Prompts**: 프롬프트 입력을 구조화하고 템플릿화하는 `PromptTemplate`. 답변에 유용한 예시를 사전에 주입해 주는 Few-Shot 프롬프팅 등을 기본 클래스로 지원합니다.
    *   **Models**: OpenAI, Anthropic, 구글 등 다양한 벤더의 채팅 API를 단일 인터페이스(`ChatOpenAI` 등)로 래핑합니다.
    *   **Output Parsers**: LLM이 반환하는 비정형 텍스트를 구조화된 형태(예: JSON, Pydantic 모델, 리스트)로 파싱하여 후처리 로직에 안전하게 바인딩할 수 있도록 돕습니다.
*   **Retrieval**:
    *   **Document Loaders**: PDF, Markdown, Notion, 구글 드라이브, 유튜브 자막 등 세상의 거의 모든 데이터 원천을 파싱하여 정형화된 `Document` 객체로 로드합니다.
    *   **Text Splitters**: 컨텍스트 윈도우의 한계와 임베딩의 유사도 정밀성을 확보하기 위해 문서를 적절한 크기(예: $500$~$1000$자 내외)의 의미 있는 청크로 분할합니다.
    *   **Vector Database & Retrievers**: 변환된 벡터 임베딩을 저장하고 검색 엔진 규격을 만들어 고속 코사인 유사도 검색을 수행합니다.
*   **Memory**:
    *   기본적으로 LLM API는 이전 대화의 맥락을 기억하지 못하는 무상태성(Stateless)을 가집니다. LangChain은 `ConversationBufferMemory`나 대화를 요약해서 넘기는 `ConversationSummaryMemory`를 통해 대화 이력을 자동으로 컨텍스트에 덧붙여 전송하는 기능을 추상화하여 지원합니다.

## 3. Reranker를 활용한 RAG의 한계 돌파와 최적화

RAG 아키텍처를 도입했음에도 불구하고 현업에서 "답변 퀄리티가 낮다"는 피드백이 쏟아지는 주된 원인은 **'임베딩 기반 유사도 검색의 한계'에** 있습니다.

### 1) 단순 벡터 유사도 검색의 병목 현상
우리가 벡터 DB에서 질문 벡터 $\mathbf{q}$와 문서 벡터 $\mathbf{d}$의 코사인 유사도를 연산하여 상위 문서를 가져올 때(Bi-Encoder 방식), 각 벡터는 상대방의 존재를 모르는 상태에서 독립적으로 압축된 벡터 공간 내 거리만 계산합니다. 이 방식은 검색 속도가 비약적으로 빠르지만, 단어나 문맥의 미묘한 의미 차이를 완벽히 반영하지 못해 질문과 전혀 다른 맥락의 문서 청크가 Top-K 결과에 섞여 들어오기 쉽습니다.

### 📊 검색 정밀도의 차이: Bi-Encoder와 Cross-Encoder의 수학적 대조
임베딩 기반 검색과 리랭커의 본질적인 정확도 차이는 모델 구조에 기인합니다.

*   **Bi-Encoder (임베딩 검색):** 
    두 텍스트를 독립적으로 인코딩하여 고정 차원 벡터 공간에 투영한 뒤 코사인 유사도를 연산합니다. 
    $$\text{Similarity}(q, d) = \cos(\mathbf{E}(q), \mathbf{E}(d)) = \frac{\mathbf{E}(q) \cdot \mathbf{E}(d)}{\|\mathbf{E}(q)\| \|\mathbf{E}(d)\|}$$
    이 방식은 단어나 문장의 토큰 레벨 어텐션(Attention) 연산이 전혀 발생하지 않아 문장의 디테일한 부정어("않다", "없다")나 미묘한 어순 관계를 놓치기 쉽습니다.
*   **Cross-Encoder (Reranker):**
    질문(Query)과 문서(Document)를 하나의 쌍으로 묶어 단일 인코더 모델(BERT 등)의 입력으로 주입합니다.
    $$\text{Score}(q, d) = \text{MLP}(\text{Transformer}([q; d]))$$
    이 방식은 질문의 모든 토큰과 문서의 모든 토큰 간의 **교차 어텐션(Cross-Attention)을** 직접 계산하므로 연산량은 현저히 높지만 극도로 높은 문맥 정합성 스코어를 산출해 냅니다.

따라서 1차로 속도가 빠른 Bi-Encoder 검색을 통해 후보군(Top-20)을 걸러내고, 2차로 정교한 Cross-Encoder Reranker(예: Cohere Rerank, BGE-Reranker)를 사용하여 최종 Top-3 문서를 압축해 전달하는 '하이브리드 파이프라인'이 필수적입니다.

### 2) Reranker를 통한 검색 실패 대응과 에이전틱 분기
만약 Reranker 단계에서 모든 문서 후보군이 컷오프(임계치 미달)되어 **"유사한 문서를 전혀 찾지 못했을 때"** RAG 시스템은 오답을 방지하기 위한 대안 행동(Fallback)을 취해야 합니다.

*   **Query Translation (질문 번역/확장):** 최초 질문을 LLM을 사용하여 더 명확하고 구체적인 키워드로 재생성(Query Rewriting)하거나, 의미가 유사한 3개 버전의 다중 질문으로 확장하여 다시 Retrieval-Rerank 사이클을 가동합니다.
*   **Fallback Search**: 사내 데이터나 기존 지식 베이스에 문서가 없을 경우, 에이전트가 인터넷 검색 도구(Tavily API 등)를 트리거하여 외부 웹 데이터를 임시 Context로 수집합니다.
*   **안전한 거절**: 검색 결과의 신뢰도가 낮을 때 억지로 생성 모델에 답변을 강제하지 않고, "죄송하지만 제공된 지식 소스 내에서 답변을 찾을 수 없습니다"라는 팩트 중심의 안전한 거절 답변을 출력하도록 템플릿 구조를 제어합니다.

## 4. LangGraph의 등장 배경과 필요성

LangChain의 LCEL은 데이터를 물 흐르듯 가공해 나가는 방향성 비순환 그래프(DAG) 구조를 만드는 데 탁월하지만, 실제 프로덕션 환경의 똑똑한 AI 에이전트는 단방향 흐름만으로 완성되지 않습니다.

### 1) LCEL (단방향 DAG)의 한계
Reranker가 검색된 문서의 적합성이 부족하다고 판정했을 때, 에이전트는 **"다시 검색 단계로 돌아가 쿼리를 바꿔서 재시도하라"는** 결정을 내려야 합니다. 그러나 LCEL은 노드 간 순환(Cyclic Loop)이 원칙적으로 금지되어 있어 이러한 되돌아가기 피드백 루프를 자연스럽게 코딩하기가 매우 어렵습니다.

### 2) Agentic RAG의 핵심, 순환(Loops)과 상태(State) 관리
LLM이 추론 과정 중 도구(Tool)를 사용해 보고, 그 결과를 스스로 검증(Self-Correction/Reflection)하여 루프를 돌기 위해서는 다음 두 가지가 보장되어야 합니다.
1.  **순환 루프(Loops):** 특정 노드의 수행 결과에 따라 분기하여 이전의 특정 단계로 무제한/조건부 회귀할 수 있는 그래프 아키텍처.
2.  **상태(State) 보존**: 루프를 돌면서 질문이 어떻게 변형되었는지, 이전 도구의 반환값은 무엇이었는지가 일관되게 누적되는 공용 메모리 스페이스.

이러한 요구 사항을 충족하기 위해 LangChain 생태계의 하위 라이브러리로 탄생한 프레임워크가 바로 **LangGraph**입니다.

## 5. LangGraph의 핵심 아키텍처와 개념

LangGraph는 상태 머신(State Machine)을 기반으로 그래프 형태의 에이전트를 정의하는 구조적 프레임워크를 제공합니다.

### ⚙️ 안정적인 흐름 제어: 왜 LangGraph의 State는 불변(Immutable)이어야 하는가?
기존의 단순 전역 변수나 무상태(Stateless) 체인과 달리, LangGraph는 그래프 내의 상태(State)를 변경할 때 반드시 **불변성(Immutability)을** 지키는 단방향 상태 갱신 기법을 취합니다.
그래프의 상태가 변경될 때마다 상태의 복사본이 생겨나며(Checkpointing), 이를 통해 우리는:
1.  **Time-travel(시간 여행):** 에이전트가 오작동했을 때, 문제 발생 직전의 State 체크포인트로 돌아가 해당 시점의 입력값을 복구하고 재실행할 수 있습니다.
2.  **Human-in-the-loop(인간 개입):** LLM이 도구(예: 이메일 전송, 데이터 삭제)를 사용하기 전, 상태를 일시정지(`Interrupt`)해 두고 인간의 승인을 받아 다음 노드로 전달할 수 있는 오케스트레이션이 가능해집니다.

이 상태 데이터 구조는 파이썬 딕셔너리 기반의 스키마로 정의되며, 여러 노드에서 동시에 동일한 상태 필드를 조작할 때 발생하는 충돌을 방지하기 위해 `Operator`를 이용한 합산(Reducers, 예: `operator.add`) 연산 규칙을 제공합니다.

### 1) Nodes & Edges
*   **Nodes**: 실제 파이썬 함수나 컴포넌트 단위로, 상태를 입력받아 상태를 변경하여 반환합니다.
*   **Edges**: 노드가 완료되었을 때 무조건 다음 노드로 흐름을 넘겨주는 고정 통로입니다.
*   **Conditional Edges (조건부 엣지):** 현재 상태값(예: Reranker 점수나 LLM의 판단 결과)을 기반으로 다음에 실행할 노드를 동적으로 평가하여 분기합니다.

### 2) Recursion Limit (재귀 한계)
루프가 포함된 그래프의 특성상 에이전트가 무한 루프(Infinite Loop)에 빠져 API 비용을 과도하게 소진하거나 멈추는 상황이 발생할 수 있습니다.
LangGraph는 이를 제어하기 위해 실행할 수 있는 최대 엣지 이동 수 단위를 정의하는 `recursion_limit`을 제공합니다.
예를 들어, 최대 20번의 루프를 타이트하게 허용하고 싶다면 아래와 같이 엣지 단위를 연산하여 제한값을 선언합니다.

```python
# 무한 루프 방지를 위한 재귀 한계 설정 (1회 이터레이션 당 약 3개 엣지 이동 기준)
config = RunnableConfig(recursion_limit=2 + 3 * 20 + 3, configurable={"thread_id": "session_1"})
```

## 6. RAG, LangChain, LangGraph의 차이 비교

세 가지 핵심 기술이 어떤 차이를 가지며 상호 작용하는지 아래 표로 정리하였습니다.

| 비교 항목 | RAG (검색 증강 생성) | LangChain (랭체인) | LangGraph (랭그래프) |
| :--- | :--- | :--- | :--- |
| **개념 정의** | 모델 외부에 문서를 추가해 답변율을 높이는 **아키텍처 패턴** | LLM 관련 컴포넌트를 조립하는 **오케스트레이션 프레임워크** | 순환(Loop)과 상태를 유지하며 에이전트를 짜는 **그래프 라이브러리** |
| **제어 흐름** | 일방향 (검색 -> 생성) | 일방향 비순환 그래프 (LCEL DAG 구조) | 순환 구조 (Loops, 에이전트 자체 의사결정 루프) |
| **상태 관리** | 별도로 처리하지 않음 | 단기/장기 메모리 연동 수준 | 공유 State 스키마 기반 실시간 상태 관리 및 체크포인팅 |
| **에이전트 구현** | 불가 (단방향 텍스트 주입에 한함) | 기초적인 단방향 에이전트 런타임 지원 | 고도화된 멀티 에이전트 워크플로우 구현에 최적화 |
| **한계 및 대안** | 벡터 검색 실패 시 오답 노출 ➔ Reranker 필수 | 순환 루프 제어 불가 ➔ LangGraph로 확장 | 높은 설계 복잡도, JSON 직렬화 한계 등 ➔ 면밀한 스키마 설계 요구 |
