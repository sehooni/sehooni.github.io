## Introduction

최근 LLM(대형 언어 모델) 기술의 비약적인 발전에도 불구하고, 모델들은 고질적인 세 가지 한계점에 직면해 있습니다.
1.  **지식의 최신성 한계 (Knowledge Cutoff):** 모델이 사전에 학습한 시점 이후의 최신 정보를 반영할 수 없습니다.
2.  **환각 현상 (Hallucination):** 사실이 아닌 내용을 마치 사실인 것처럼 그럴싸하게 생성해 냅니다.
3.  **내부 데이터 접근 불가**: 기업이나 개인의 민감한 비공개 보안 데이터를 학습 과정에 포함시키기 어렵습니다.

이러한 문제들을 해결하기 위해 등장한 기술이 바로 **RAG (Retrieval-Augmented Generation, 검색 증강 생성)입니다**. RAG는 LLM이 답변을 생성하기 전에 신뢰할 수 있는 외부 지식 베이스(외부 문서 등)에서 관련성 높은 정보를 먼저 '검색(Retrieval)'해 오고, 그 검색 결과를 Context로 제공하여 답변을 '생성(Generation)'하게 함으로써 답변의 정확도와 실시간성을 극대화합니다.

쉽게 비유하자면, 모델의 자체 지식에만 의존해 시험을 치던 **"폐쇄형 시험(Closed-book Exam)"** 방식에서, 모르는 문제가 나오면 교과서나 참고서를 찾아보고 답하는 **"오픈북 시험(Open-book Exam)"** 방식으로 전환하는 것과 같습니다.

## 1. RAG vs Fine-Tuning: 무엇이 다를까?

외부 데이터를 LLM에 학습시키고 활용하는 또 다른 대표적인 방법으로 **파인튜닝(Fine-Tuning, 미세조정)이** 있습니다. RAG와 파인튜닝은 상호 보완적인 기술이지만, 적용 목적과 장단점이 뚜렷하게 나뉩니다.

*   **파인튜닝**: 모델의 파라미터(가중치) 자체를 직접 업데이트하여 도메인 특화 말투, 형식, 추론 스타일을 주입하는 작업입니다. 모델의 내장 지식 자체를 변화시키므로 비용과 시간이 많이 듭니다.
*   **RAG**: 모델 자체는 그대로 두고, 프롬프트에 외부 지식(Context)을 동적으로 주입하여 답변하게 만듭니다. 지식 업데이트가 매우 간편하며 투명한 출처 제시가 가능합니다.

| 비교 항목 | RAG (검색 증강 생성) | Fine-Tuning (미세조정) |
| :--- | :--- | :--- |
| **모델 가중치 업데이트** | 업데이트 안 함 (외부 Context 활용) | 직접 업데이트 (학습 필요) |
| **구현 및 유지비용** | 상대적으로 매우 저렴함 | 인프라 및 학습 비용이 매우 높음 |
| **최신 정보 반영 속도** | 즉시 반영 가능 (DB 업데이트) | 모델 재학습 전까지 반영 불가 |
| **환각 현상 제어** | 검색된 팩트를 참조하므로 매우 낮음 | 내재된 지식에 의존하므로 발생 가능 |
| **출처 제시 (Traceability)**| 정확한 문서 출처 및 인용구 제시 가능 | 출처를 추적 및 제시할 수 없음 |
| **주요 목적** | 최신 지식 확보 및 정확한 팩트 전달 | 특정 도메인의 말투 및 형식, 태스크 특화 |

## 2. RAG의 6단계 핵심 프로세스

RAG 시스템은 다음과 같은 6단계 파이프라인을 거쳐 사용자에게 정확한 답변을 제공합니다.

```
[문서 수집/정제] ──> [청킹 & 임베딩] ──> [벡터 DB 인덱싱]
                                             │ (저장)
[사용자 질문 입력] ──> [질문 임베딩] ───> [유사도 기반 검색] ──> [LLM 답변 생성]
```

### 1) Document Collection & Preprocessing (문서 수집 및 전처리)
PDF, Word, Markdown, HTML 등 다양한 형식의 원천 데이터를 읽어와 텍스트로 추출합니다. 이 단계에서 표(Table)나 이미지 내 텍스트, 불필요한 메타데이터(헤더/푸터 등)를 정제하여 데이터 노이즈를 줄이는 작업이 중요합니다.

### 2) Chunking & Embedding (청킹 및 임베딩)
*   **청킹 (Chunking):** 긴 문서 전체를 한 번에 LLM에 전달할 수 없으므로(컨텍스트 윈도우 한계 및 비용 고려), 데이터를 의미 있는 단위(예: $300$~$1000$자)로 쪼갭니다.
*   **임베딩 (Embedding):** 쪼개진 텍스트 청크를 기계가 연산할 수 있는 고차원 벡터(실수 배열)로 변환합니다. 변환된 벡터들은 의미적 유사도가 높을수록 벡터 공간 내에서 가깝게 위치하게 됩니다.

두 임베딩 벡터 $\mathbf{u}$와 $\mathbf{v}$ 사이의 유사도를 판단하기 위해 가장 널리 쓰이는 계산 공식 중 하나는 다음과 같은 **코사인 유사도(Cosine Similarity)입니다**.

$$\text{Cosine Similarity}(\mathbf{u}, \mathbf{v}) = \frac{\mathbf{u} \cdot \mathbf{v}}{\|\mathbf{u}\| \|\mathbf{v}\|} = \frac{\sum_{i=1}^{n} u_i v_i}{\sqrt{\sum_{i=1}^{n} u_i^2} \sqrt{\sum_{i=1}^{n} v_i^2}}$$

### 3) Vector Indexing (벡터 인덱싱 및 Vector DB)
임베딩이 완료된 벡터 데이터와 각 벡터에 매칭되는 실제 텍스트 정보(메타데이터)를 **Vector Database (예: Pinecone, Milvus, Chroma, FAISS)에** 저장합니다. 이때 대규모 벡터 공간에서 빠르게 검색하기 위해 HNSW(Hierarchical Navigable Small World) 등과 같은 고성능 벡터 인덱싱 알고리즘이 적용됩니다.

### 4) Query Processing (사용자 질문 처리)
사용자가 질문(Query)을 입력하면, 문서들을 임베딩할 때 사용했던 것과 **동일한 임베딩 모델을** 거쳐 사용자 질문을 실수 벡터로 변환합니다.

### 5) Retrieval & Ranking (검색 및 랭킹)
*   **검색**: 사용자 질문 벡터와 Vector DB에 저장되어 있던 문서 청크 벡터들 간의 코사인 유사도를 계산하여 가장 연관성이 높은 상위 $K$개의 문서 청크를 추출합니다.
*   **랭킹(Reranking):** 벡터 유사도 계산은 빠르지만 미세한 맥락 파악에 약할 수 있습니다. 최근에는 1차 검색 결과를 Cross-Encoder 계열의 리랭커(Reranker) 모델로 재평가 및 정렬하여 가장 유의미한 문서만 필터링하는 파이프라인이 필수화되었습니다.

### 6) Generation (LLM 답변 생성)
검색된 고품질의 문서 청크(Context)들을 사용자 질문과 함께 하나의 템플릿(System Prompt)에 결합하여 LLM에 최종 주입합니다. LLM은 주입받은 Context 내부의 팩트에 기반하여 왜곡 없는 신뢰성 있는 답변을 생성하게 됩니다.

## 3. Python 코드로 구현해보는 간단한 RAG 원리

Numpy를 활용하여 질문 벡터와 문서 벡터들 간의 유사도를 연산하고 가장 매칭율이 높은 정보를 매핑하는 원리를 텍스트 임베딩 모델 과정 없이 러프하게 모방한 기초 예제입니다.

```python
import numpy as np

# 코사인 유사도 함수 정의
def cosine_similarity(u, v):
    return np.dot(u, v) / (np.linalg.norm(u) * np.linalg.norm(v))

# 1. 쪼개진 문서 데이터셋 (Database metadata)
documents = [
    "AlphaFold2는 딥러닝 기반으로 단백질 3차원 구조를 혁신적으로 예측하는 모델이다.",
    "RAG(검색 증강 생성)는 LLM이 답변하기 전 외부 문서에서 정보를 검색해오는 기술이다.",
    "파인튜닝은 데이터셋을 이용하여 거대 언어 모델의 가중치를 직접 미세 조정하는 작업이다."
]

# 2. 각 문서들의 모방 임베딩 벡터 (실제 임베딩 모델의 차원은 대게 768, 1536 등 고차원입니다)
doc_embeddings = [
    np.array([0.8, 0.1, 0.1]),  # 바이오/단백질 관련 성분이 높은 공간
    np.array([0.2, 0.8, 0.2]),  # LLM/검색 기술 관련 성분이 높은 공간
    np.array([0.1, 0.3, 0.8])   # 모델 미세조정/파라미터 학습 성분 공간
]

# 3. 사용자 질문 벡터 (RAG의 동작 원리에 대한 질문 모방)
query_embedding = np.array([0.15, 0.85, 0.18])

# 4. 각 문서 벡터들과의 유사도 계산
similarities = [cosine_similarity(query_embedding, doc_vec) for doc_vec in doc_embeddings]

# 5. 가장 유사도가 높은 문서 탐색
best_index = np.argmax(similarities)
print(f"사용자 질문에 대해 검색된 1순위 문서:\n-> \"{documents[best_index]}\"")
print(f"매칭 코사인 유사도 점수: {similarities[best_index]:.4f}")
```

## 4. Advanced RAG: 더 똑똑한 검색을 위해

가장 기본적인 Naive RAG의 경우, 임베딩 모델의 한계나 데이터 품질 저하로 인해 "잘못된 문서 검색으로 인한 오답 생성" 문제를 일으키기도 합니다. 이를 위해 현대 AI 아키텍처에서는 다음과 같은 **Advanced RAG** 기법들이 적용됩니다.

*   **Hybrid Search (하이브리드 검색):**
    키워드 매칭 중심의 전통적인 검색 알고리즘(예: BM25)과 문맥 유사성 중심의 밀집 벡터 검색(Dense Retrieval)을 적절한 가중치(RRF 등)로 결합하여 검색 성공률을 극대화합니다.
*   **Query Expansion (질문 확장):**
    사용자가 질문을 짧고 모호하게 작성했을 때, LLM이 이를 구체적이고 전문적인 여러 개의 하위 질문으로 확장/재작성(Rewriting)하여 다각도로 검색을 수행하는 기법입니다.
*   **Agentic RAG (에이전트형 RAG):**
    단순 1회성 검색에 그치지 않고, 에이전트가 "가져온 외부 데이터가 질문에 대답하기에 충분한가?"를 스스로 평가하고 부족하다면 검색 쿼리를 수정해 재검색하는 등 closed-loop를 도는 고도화된 아키텍처입니다.

## 맺음말

RAG는 LLM을 실생활에 쓸모 있고 정확한 프로덕트로 전환하기 위한 가장 실용적인 해답 중 하나입니다. 이번 포스팅을 통해 RAG의 개념적 차별성부터 6단계의 세부 아키텍처, 그리고 실제 유사도 비교 계산 원리까지 살펴보았습니다.
